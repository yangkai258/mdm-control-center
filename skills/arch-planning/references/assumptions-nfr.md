# 假设清单 + NFR 量化门槛

## 关键假设分级

每个架构设计基于一组假设，明确列出并标出确信度：

| 假设 | 确信度 | 如果错了的应对 |
|------|--------|--------------|
| 日活用户半年内不超过 10 万 | 高 | 水平扩展 |
| 第三方支付 API SLA ≥ 99.9% | 中 | 熔断 + 降级方案 |
| 团队 3 个月内熟悉新框架 | 低 | 选更熟悉的方案，或加培训成本 |

**规则**：低确信假设必须触发后续验证动作（POC / 压测 / 备选方案）

---

## 确信度速判法（3行判断）

| 确信度 | 判断标准 |
|--------|---------|
| **高** | 团队亲手做过、官方文档明确支持、有现成生产案例 |
| **中** | 理论上可行，但没亲手验证过；有不确定的边界条件 |
| **低** | 纯推测、依赖外部未承诺的特性、与团队现有能力有明显 gap |

**低确信假设必须立即触发 POC 或备选方案，不能拖。**

没有数字就无法验收。关键 NFR 必须写具体阈值：

### 性能
| 指标 | 阈值 |
|------|------|
| 读接口 P99 | < 200ms | 在 500 QPS、数据量 5000 万条、缓存命中率 80% 的条件下 |
| 写接口 P99 | < 500ms | 在 200 QPS、数据量 1 亿条的条件下 |
| 首页加载时间 | < 3s（3G 网络） | 包含首屏渲染 + API 首次返回 |

### 可用性
| 指标 | 阈值 |
|------|------|
| 年可用率 | ≥ 99.9%（允许计划外故障 < 8.7h/年） |
| 故障恢复时间（MTTR） | < 30min |
| 允许同时故障节点数 | 1 |

### 数据
| 指标 | 阈值 |
|------|------|
| 数据丢失（RPO） | 0（不允许丢数据） |
| 备份频率 | 每小时增量，每天全量 |

### 安全
| 指标 | 阈值 |
|------|------|
| 密码存储 | bcrypt + salt，不可逆 |
| 敏感数据 | 必须加密存储 |
| 认证失败锁定 | 5 次后锁定 15min |
